# 技术可行性分析报告（老人健康管理系统·MVP）

日期：2026-04-15

## 1. 项目结论
- MVP完全可实现：微信小程序（老人/家属）+ 医生Web后台 + Python后端 + MQTT数据接入 + 规则阈值报告 + 未体检提醒闭环
- 关键可行性点：MQTT幂等入库、ECG大数据存储、权限与隐私合规、定时任务去重、订阅消息授权与降级

## 2. MVP范围确认
- 小程序：openid登录、绑定手机号、资料完善、报告列表/详情、通知中心、家属绑定（只读）
- 医生Web：用户档案CRUD、报告查看、未体检待办（3/6个月）处理与触达记录、审计查看（建议）
- 后端：鉴权/权限、数据模型、MQTT订阅验签与幂等、会话聚合、规则阈值引擎、报告生成、定时提醒、站内通知、订阅消息发送记录（可开关mock）
- ECG：分片不完整时不生成报告
- 血糖：`GLU_FASTING`与`GLU_RANDOM`两种编码

## 3. 技术方案选型（1~3套）

### 3.1 后端框架
- 方案A（推荐）：FastAPI + Uvicorn
  - 优点：性能好、类型友好、接口文档自动生成、适合服务化与异步任务扩展
  - 风险：需要团队熟悉Pydantic/依赖注入风格
- 方案B：Django + DRF
  - 优点：生态成熟、后台管理与权限体系强、开发规范化
  - 风险：相对厚重，MQTT接入与异步任务需要更明确拆分
- 方案C：Flask
  - 优点：轻量灵活
  - 风险：工程化与规范需要自行补齐，MVP后期维护成本更高

### 3.2 数据库
- 方案A（推荐）：PostgreSQL
  - 优点：JSON字段、索引能力强，适合报告content_json与审计detail_json
- 方案B：MySQL 8
  - 优点：团队常见、运维成熟
- 结论：两者均可；若团队更熟MySQL则优先MySQL 8

### 3.3 ECG大数据存储
- 方案A（推荐）：对象存储（MinIO/OSS/S3）+ 数据库索引
  - 优点：成本低、扩展性强、避免数据库膨胀
- 方案B：数据库分片表存二进制/大字段
  - 风险：数据库膨胀、备份与性能压力大

### 3.4 异步任务
- 方案A（推荐）：RQ/Celery/Arq 任一 + Redis
  - 用途：报告生成、订阅消息发送、错误重放、定时扫描
- 方案B：仅用数据库表 + 定时轮询worker（MVP可用）
  - 风险：并发扩展与重试策略需要更仔细实现

### 3.5 MQTT接入
- 方案A（推荐）：后端服务内置MQTT订阅消费者（paho-mqtt等）
  - 优点：链路短、实现快
  - 风险：与HTTP服务同进程需隔离故障与重连策略
- 方案B：MQTT桥接→HTTP Ingest API
  - 优点：隔离协议层，扩展更好
  - 风险：系统复杂度提升

## 4. 关键技术难点与应对
- 幂等与乱序：`deviceId+msgId`唯一；session聚合允许补传；重复包返回成功但不重复入库
- 防重放：引入`sentAt`（发送时间）做5分钟窗口校验；`ts`保留业务采集时间
- ECG分片：基于`ecg_meta.totalChunks`校验缺片；缺片则会话标记partial且不生成报告
- 权限防越权：报告详情按report归属user二次校验，避免reportId猜测越权
- 合规与隐私：身份证加密存储、脱敏展示、医生端审计留痕、订阅消息需用户授权

## 5. 风险评估（成本与风险）
- 需求风险：指标阈值与文案需要医生确认，建议上线前做一次规则审阅
- 合规风险：健康数据属于敏感信息，需明确隐私政策、数据留存与访问审计
- 对接风险：硬件侧协议对齐（MQTT payload、签名、分片策略）需要测试向量与联调环境

## 6. 研发分工建议（MVP）
- 前端（小程序）：登录/绑定/资料/首页/报告/通知/家属绑定页面与联调
- 前端（Web）：医生后台登录、用户CRUD、报告查看、待办处理、审计展示
- 后端：鉴权、数据模型、MQTT接入幂等、报告引擎、提醒任务、通知与订阅消息记录、审计
- 测试：按验收用例覆盖幂等、防越权、ECG缺片不出报告、3/6个月待办去重

## 7. MVP验收闭环（必须跑通）
- 模拟MQTT消息：start→vitals→end → 生成报告 → 小程序可查看
- 3/6个月未体检：生成医生待办 → 医生处理 → 小程序通知中心出现提醒记录
- 家属绑定：申请→老人确认→家属只读查看老人报告（解绑后立即403）
