## 1. 背景与问题
- 老人健康体检不连续、数据难汇总，医生无法及时发现风险并触达，家属缺少稳定查看通道
- 设备侧可采集生命体征与心电图等数据，但云端缺少统一接入、幂等入库、分析生成报告与提醒闭环

---

## 2. 产品目标与成功指标

### 2.1 产品目标（MVP）
- 形成“设备数据接入 → 入库 → 规则分析 → 报告生成 → 小程序可查看 → 未体检提醒医生处理”的闭环
- 小程序完成：openid登录 + 绑定手机号 + 资料完善 + 报告/通知查看 + 家属绑定（只读）
- 医生端完成：用户档案CRUD + 报告查看 + 未体检提醒待办处理与触达记录

### 2.2 成功指标（MVP观测口径）
- 账号闭环：openid登录→手机绑定成功率
- 报告闭环：报告生成数、报告查看率、订阅消息发送成功率（有授权的前提下）
- 医生效率：未体检待办处理率、平均处理耗时
- 连续体检：3个月/6个月未体检人数趋势

### 3.1 角色定义

## 4. 业务范围（做/不做）

### 4.1 MVP做
- 小程序：openid登录、绑定手机（短信验证码）、资料完善、报告列表/详情、通知中心、家属绑定（只读）
- 医生Web：用户档案CRUD、体检会话与报告查看、未体检提醒待办（3/6个月）与触达记录
- 后端：账号/权限、数据模型、MQTT协议与topic规范、幂等入库、规则阈值分析、报告结构化输出、定时提醒任务、通知记录、审计
- ECG大数据：采用对象存储或分片表方案（数据库仅存索引与元信息）

### 4.2 MVP不做（后续迭代）
- AI模型分析、医生逐条审核后发布报告（MVP默认自动生成）
- 复杂机构/科室数据权限（MVP可先单机构全量可见）
- 设备端完整固件实现细节（MVP以接口契约+模拟消息为主）
- 小程序完整ECG波形绘制（MVP可先展示摘要与下载/查看占位）

## 5. 总体方案与关键约束

### 5.1 总体方案
- 设备侧通过 MQTT 上行数据到云端（先定义协议），云端接入服务订阅并解析
- 后端将数据按“体检会话”聚合，生成结构化指标与ECG索引
- 规则阈值引擎输出异常项与建议，生成报告并推送通知
- 定时任务扫描未体检阈值，生成医生待办，由医生端触达用户

### 5.2 关键约束
- 幂等：重复上报不产生重复数据/重复报告
- 合规：手机号/身份证/健康数据必须加密存储、脱敏展示、访问审计
- 大数据：ECG等不直接存数据库大字段，走对象存储或分片存储
- 网络不稳定：支持离线缓存与补传（first-control/senor侧）

## 6. 核心业务流程（端到端）

### 6.1 流程一：体检数据→报告→用户查看
1. 设备产生数据（vitals/ecg）并通过 MQTT 上报（含msgId、sessionId）
2. 云端接入服务校验签名/鉴权，写入接入表（幂等）
3. 数据入库：写入measurements/ecg_files等
4. 收到 `session_end` 或满足会话完成条件：触发报告生成
5. 生成 `reports` 并创建通知 `notifications`
6. 小程序用户查看报告列表/详情

### 6.2 流程二：未体检提醒（3个月/6个月）
1. 每日定时任务扫描：`now - last_exam_time`
2. 满足>=3个月生成普通待办，>=6个月生成高优先级待办
3. 医生端查看待办并处理（触达用户/记录备注/标记完成）
4. 用户端收到通知或在通知中心看到提醒记录

## 7. 功能需求

### 7.1 小程序端（老人/家属）

#### 7.1.1 登录与绑定手机号
- 登录方式：openid登录 + 绑定手机号（短信验证码）
- 绑定前访问限制：禁止查看报告详情与通知详情（可显示引导页）
- 绑定成功后：可完善资料、查看报告与通知、进行家属绑定

**异常/边界**
- 验证码错误/过期：提示并可重新获取（限制频率）
- 手机号已绑定其他openid：提示“手机号已被占用”，提供申诉/联系客服入口（MVP可先提示）

#### 7.1.2 个人资料完善
- 必填：手机号（已验证）
- 选填：姓名、性别、出生日期、身份证号（若启用必须加密存储）
- 展示规则：身份证号仅显示前2后2，其余用`*`脱敏

#### 7.1.3 家属绑定（只读）
- 目标：家属可只读查看老人报告与通知
- 推荐绑定流程（MVP）：家属发起绑定→老人端确认授权（弹窗/验证码确认）→建立关系
- 权限：默认只读；解绑需双方任一方操作并二次确认

#### 7.1.4 报告与体检记录
- 报告列表：时间、结论摘要、风险等级、异常项数量
- 报告详情：
  - 体检信息：体检时间、来源设备、会话ID
  - 指标明细：按分类展示（基础生命体征、血糖/体重等）
  - 异常项：置顶展示（含解释与建议）
  - ECG：展示摘要（时长、采样率、结论）；波形展示后续迭代

#### 7.1.5 通知中心
- 类型：报告生成、未体检提醒、系统通知
- 支持：未读红点、标记已读、进入详情
- 订阅消息：用户授权后可触达（失败记录状态）

---

### 7.2 医生Web管理端

#### 7.2.1 用户档案管理（CRUD）
- 列表：按姓名/手机号/身份证后四位搜索；支持分页
- 详情：基本信息、绑定设备（如有）、体检记录、报告列表、提醒记录
- 增删改：全部写审计日志

#### 7.2.2 报告查看
- 支持按用户/时间筛选报告
- 报告详情含：异常项、规则命中详情、建议、报告版本

#### 7.2.3 未体检提醒待办
- 待办列表：按阈值类型（3个月/6个月）、状态、用户筛选
- 处理动作：选择触达方式（站内/订阅消息记录）、填写备注、标记完成
- 处理后：生成通知记录并关联到用户

---

### 7.3 后端（Python服务）

#### 7.3.1 鉴权与会话
- 小程序：token（openid维度）
- 医生端：token（doctor维度）
- 所有敏感资源访问写审计（至少记录医生端）

#### 7.3.2 数据接入与幂等
- 使用 MQTT 订阅 topic
- 每条消息必须包含 `msgId`，服务端以 `deviceId+msgId` 做唯一幂等
- 解析失败进入 `ingest_errors`，可重放

#### 7.3.3 报告生成服务
- 触发：收到`session_end` 或会话满足“完成条件”
- 输出：结构化 `content_json`（见第9章）
- 版本：报告写入`rule_version`，可追溯

#### 7.3.4 定时任务（未体检提醒）
- 每日任务生成待办：3个月/6个月
- 避免重复：对同一用户、同一阈值类型、同一周期窗口只生成1条未完成待办

## 8. 数据与接口契约

### 8.1 MQTT Topic规范（MVP）
- 上行（设备→云）：
  - `fc/{fcId}/up`
  - `watch/{watchId}/up`
  - `sensor/{sensorId}/up`
- 下行（云→设备，可选）：
  - `fc/{fcId}/down`
  - `watch/{watchId}/down`

### 8.2 MQTT QoS/Retain建议
- QoS：上行QoS1（至少一次）；下行按指令重要性QoS1
- Retain：默认不retain

### 8.3 MQTT通用Payload（JSON）
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| msgId | string | ✅ | 幂等ID（UUID） |
| ts | int64 | ✅ | 毫秒时间戳 |
| deviceId | string | ✅ | 设备唯一ID |
| deviceType | string | ✅ | `fc`/`watch`/`sensor` |
| sessionId | string | ⛳ | 体检会话ID（强烈建议） |
| type | string | ✅ | 消息类型（见8.4） |
| data | object | ✅ | 业务数据体 |
| ver | string | ⛳ | 协议版本，如`1.0` |
| sig | string | ✅ | 签名（HMAC或证书签名） |

### 8.4 MQTT消息类型枚举（MVP）
- `device_meta`：设备元信息上报
- `session_start`：体检会话开始
- `vitals`：结构化生命体征/血糖/体重等
- `ecg_meta`：ECG文件/流元信息（采样率、时长等）
- `ecg_chunk`：ECG分片数据
- `session_end`：体检会话结束（触发报告生成）
- `error`：设备侧错误上报（可选）

### 8.5 各类型data字段表与示例

#### 8.5.1 device_meta
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| model | string | ✅ | 型号 |
| fwVersion | string | ✅ | 固件版本 |
| capabilities | string[] | ✅ | 能力集，如`["HR","BP","SPO2","ECG"]` |
| ip | string | ⛳ | IP（如有） |
| rssi | number | ⛳ | 信号强度（如有） |

### 8.6 指标编码（MVP建议）
- `HR`：心率（bpm）
- `SPO2`：血氧（%）
- `TEMP`：体温（℃）
- `BP_SYS`：收缩压（mmHg）
- `BP_DIA`：舒张压（mmHg）
- `GLU`：血糖（mmol/L）
- `WT`：体重（kg）

---

### 8.7 接入错误码（MVP）
| 场景 | 错误码 | 说明 | 处理建议 |
|---|---|---|---|
| 签名校验失败 | ING_401_SIG | `sig`不合法 | 丢弃并记录错误；设备侧检查密钥 |
| 缺少必填字段 | ING_400_FIELD | 缺`msgId`/`type`等 | 丢弃并记录；设备修复 |
| 幂等冲突 | ING_200_DUP | 重复`deviceId+msgId` | 直接返回成功（不重复入库） |
| 会话不存在 | ING_404_SESS | `sessionId`未找到且无start | 创建会话或记录待补全 |
| 分片不完整 | ING_409_CHUNK | ECG分片缺失/校验失败 | 标记待补传并告警 |
| 服务异常 | ING_500 | 解析/入库失败 | 记录错误+可重试 |

## 9. 规则阈值与报告结构

### 9.1 规则阈值引擎（MVP）
- 输入：用户基础信息（年龄/性别）+ 指标数据（measurements）+ ECG摘要（可选）
- 输出：风险等级、异常项列表、建议文案、规则命中详情
- 版本：每次规则阈值/文案调整递增 `rule_version`

### 9.2 报告content_json结构定义（前端渲染契约）
`reports.content_json`建议结构：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| schemaVersion | string | ✅ | 如`1.0` |
| reportId | string | ✅ | 报告ID |
| sessionId | string | ✅ | 会话ID |
| generatedAt | int64 | ✅ | 毫秒时间戳 |
| user | object | ✅ | 用户摘要 |
| summary | object | ✅ | 总结区 |
| sections | object[] | ✅ | 模块化展示区 |
| abnormalities | object[] | ✅ | 异常项（置顶） |
| suggestions | string[] | ✅ | 建议列表 |
| ruleHits | object[] | ✅ | 规则命中（医生端可见，用户端可隐藏） |

`user`：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| name | string | ⛳ | 姓名 |
| gender | string | ⛳ | `M`/`F` |
| age | int | ⛳ | 年龄 |

`summary`：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| riskLevel | string | ✅ | `low`/`medium`/`high` |
| text | string | ✅ | 摘要 |
| abnormalCount | int | ✅ | 异常项数量 |

`sections[]`：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| key | string | ✅ | `vitals`/`ecg`/`glucose_weight`等 |
| title | string | ✅ | 模块标题 |
| items | object[] | ✅ | 指标项 |

`sections[].items[]`：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| code | string | ✅ | 指标编码 |
| name | string | ✅ | 指标名称 |
| value | number/string | ✅ | 值 |
| unit | string | ✅ | 单位 |
| level | string | ✅ | `normal`/`low`/`high`/`danger` |
| ref | string | ⛳ | 参考范围文本 |
| explain | string | ⛳ | 解释（用户可读） |

`abnormalities[]`：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| code | string | ✅ | 指标编码 |
| level | string | ✅ | `low`/`high`/`danger` |
| title | string | ✅ | 异常标题 |
| detail | string | ✅ | 异常说明 |
| advice | string | ✅ | 建议 |

`ruleHits[]`（医生端展示为主）：
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| code | string | ✅ | 指标编码 |
| ruleId | string | ✅ | 规则ID |
| version | string | ✅ | 规则版本 |
| threshold | object | ✅ | 阈值信息 |
| result | object | ✅ | 命中结果 |

## 11. 验收标准（可测用例）

### 11.1 数据接入与幂等
- 重复发送相同`deviceId+msgId`：入库仅1次，报告仅生成1份
- 缺字段/签名失败：写入`ingest_errors`，不污染业务表
- ECG分片缺失：标记会话未完成，不生成报告或生成“数据不完整”提示（产品策略二选一，MVP建议不生成）

### 11.2 报告闭环
- 发送`session_start`→`vitals`→`session_end`：生成报告与通知，小程序可查看详情
- 医生端可查看报告并看到规则命中详情

### 11.3 未体检提醒闭环
- 构造用户最后体检时间为3个月前：生成普通待办
- 构造为6个月前：生成高优先级待办
- 医生处理后：待办状态变更、生成通知记录、用户通知中心可见

### 11.4 账号与权限
- 未绑定手机号：无法访问报告详情（提示绑定）
- 家属绑定后：家属仅可读老人报告，不可修改老人资料

## 12. 风险与对策
- 隐私合规：敏感字段加密+脱敏+审计；最小权限；订阅消息需授权
- MQTT重复/乱序：幂等、按session聚合、允许补传
- ECG数据量：对象存储/分片；hash校验；按会话索引
- 运维可观测：接入错误表、重放能力、关键指标告警（失败率、延迟）
